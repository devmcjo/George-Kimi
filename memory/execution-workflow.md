## 6단계 실행 워크플로우

| 단계 | 내용 | 산출물 | 완료 기준 |
|------|------|--------|----------|
| 1단계 | 계획서 수립 | `project{N}/` 계획서 (확정 버전) | 팀 리드 승인 |
| 2단계 | Redmine 하위일감 생성 | Phase별 Level 2 + WBS별 Level 3 | 계획서 WBS와 1:1 대응 |
| 3단계 | 스크립트 작성 | `project{N}/scripts/` WBS별 스크립트 | 모든 WBS에 대응 스크립트 존재 |
| 4단계 | 정합성 검증 | 검증 보고서 | 계획서↔Redmine↔스크립트 3자 일치 |
| 5단계 | 스크립트 수행 | 코드 변경 + Redmine 상태 갱신 | vcxproj별 테스트 통과 |
| 6단계 | 최종 검증 | 종료 선언 | 1~5단계 정상 수행 확인 |

---

## 핵심 원칙

1. **계획서-Redmine-스크립트 3자 정합성**: 세 가지가 하는 일이 반드시 일치
2. **WBS 단위 실행**: WBS별 → 하위일감 → 스크립트 → 코드 수정 → vcxproj별 테스트
3. **단계별 진행**: 하나의 vcxproj 테스트 완료 후 다음 vcxproj 진행
4. **상태 추적**: 스크립트 수행 상태에 따라 Redmine 일감 상태 업데이트
5. **QA 분리**: 6단계 완료 후 별도 QA 진행

---

## Redmine 일감 구조

| Level | 내용 | 제목 규칙 |
|-------|------|----------|
| Level 1 | 프로젝트 최상위 일감 | (팀 리드 지정) |
| Level 2 | Phase별 일감 | `[상위제목 약칭] Phase N: 설명` |
| Level 3 | WBS별 일감 | `[Phase N] WBS M.M: 설명` |

- **depth 제한**: 최대 3단계 (시인성 확보)
- WBS 하위(예: 4.1.1, 2.1.1)는 Level 3 일감의 description에 포함

---

## 스크립트 작성 규칙

- **위치**: `project{N}/scripts/` (또는 `x64script/` 등 작업별 디렉토리)
- **파일명**: `phase{N}-wbs{M}-{설명}.{확장자}` (예: `phase1-wbs11-dword-this.sh`)
- **내용**: 해당 WBS의 [입력] → [작업] → [출력] → [검증] 절차 구현
- **실행 환경**: 스크립트 상단에 명시 (Git Bash, PowerShell 등)
